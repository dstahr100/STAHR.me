    • Introduction

     

    A few key concepts are involved in accessible web development - The Accessibility Tree, Focus, and Semantics. Each concept is essential to understanding web accessibility. Therefore, the main ideas behind each of these concepts will be explored.   

     

    The Accessibility Tree

     

    The first important concept is The Accessibility Tree. The Accessibility Tree is essentially a modified version of the Document Object Model – but with fewer nodes. The reason why there are fewer nodes (or objects that represents a part of a webpage document) is because a lot of nodes are used only for displaying visual information.

     

    Consequently, The Accessibility Tree essentially acts as an API (application programing interface) that tells screen readers what is on a webpage. The Accessibility Tree does this by telling screen readers the semantics and affordances of an HTML element – which will be talked about later on.

     

    A good way to visualize The Accessibility Tree is to think about how a typical 1990’s website looks. They had a few carefully chosen images, plain text, and a bunch of links. It may look more aesthetically simple compared to a more modern website. However, a 1990s website nonetheless still displayed all the information that a user needed. Therefore, a 1990s website gives a pretty good idea to how a screen reader user experiences a website with The Accessibility Tree. 

     

    Without the information contained in The Accessibility Tree, screen reader users would not be able to traverse a website. Keeping The Accessibility Tree in mind is important because it allows developers to know how their projects are processed for assistive technologies.    

    Focus

     

    The next important concept is Focus. In a nutshell, Focus is directing keyboard events to a selected HTML element such as a button or a link. Keyboard users primarily use Focus to navigate through a webpage. This is because by using the Tab or the Shift + Tab keys someone can jump to an interactive HTML element and use it.       

     

    What is an interactive HTML element? An interactive HTML element is an element that requires input from a user. For example, a link requires a user to click on it in order for the link to be opened and a form requires a user to enter text.

     

    As a result, when a keyboard user hits the Tab key, they will focused on the first interactive HTML element on the webpage. Once focused, a user can then use the interactive HTML element with a keyboard event. For instance, an example of a keyboard event would be a user hitting the Enter key on a focused button to submit a completed form.

     

    The specific order that the HTML elements are focused in on a webpage is called the Tab Order. The Tab Order is dictated by the structure of a webpage’s HTML code. Therefore, the job of an accessible web coder is to make sure the tab order of a webpage is logical.

     

    Focus is important to accessible web development because it allows keyboard users to get to interactive elements. From knowing how Focus works, a web developer can better know how to design better accessible webpages.

     

    Semantics

     

    The last concept is Semantics. This concept is very important because Semantics makes up the fundamentals of verbal user interfaces for disabled users.

     

    Before diving into Semantics, it is beneficial to first understand the term Affordances. One definition of Affordances is that “Affordances offer or afford a person the opportunity to perform an action.”

     

    So what does that mean? The best way to illustrate what Affordances are is from the design of a teapot. A teapot instructs someone how to use it from its spout and from its handle. The handle indicates to someone that they can pick up the teapot and the spout indicates to someone that they can pour water out of it as well. Consequently, both the handle and the spout are Affordances for a teapot. 

     

    In web design, developers try to create these same Affordances visually so that people can intuitively use a site. For example, a developer might add a shadow to a button to make it 3D and indicate to someone that it can a pressed. However, such visual Affordances will not work for screen reader users and so another method has to be used.

     

    This is where Semantics comes into play for accessible web development. Basically, audio user interphases re-create the same visual Affordances that web developers make with Semantics. Developers can add Semantics to a HTML element by using semantical attributes. These attributes are:

     

    1.    Role

    2.    Name

    3.    State

    4.    Value

     

    The Role attribute tells The Accessibility Tree what the HTML element does. For example, if a link has been designed to act more like a button, the link’s role attribute should be changed to “button” because the element is being used in a button capacity. The next attribute is the Name attribute. The Name attribute assigns a label to a HTML element. This label is then read by a screen reader when the user interacts with the HTML element.

     

    After that, there is the State attribute that tells screen readers the current state of an HTML element. An example of the State attribute would be if a form list is expanded or not. Lastly, there is the Value attribute that defines the value of an input HTML element.

     

    Most of these semantical attributes are added by default to  semantical HTML5 elements. However, semantical attributes can also be added to non-semantical HTML5 elements as well to make an existing element more useful.

     

    If a developer wants to improve accessibility, a good place to start is to add helpful Semantics. Proper Semantics on a website allow disabled users to have an intuitively seamless experience.
