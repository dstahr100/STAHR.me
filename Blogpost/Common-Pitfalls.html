    • Introduction

     

    From my short time learning how to better implement websites that are disability accessible, I have discovered the common pitfalls that would result in a developer making a site that is not accessible to all users.

     

    I myself am guilty of not including good accessibility practices in my projects. A quick accessibility audit of my website has shown me that my site is not currently optimized for accessibility. However, I can change this by learning the correct practices and by gradually updating my website as I learn them.

     

    This blogpost will give a brief overview of the common pitfalls that leads to a developer making an inaccessible website. Some of these topics will be cover in more detail in later entries.   

     

    Lack of Knowledge

     

    The greatest barrier that prevents developers from making accessible websites is knowing on how to make them. In fact, the other categories here can all fall under the umbrella of lack of knowledge. This is because not knowing how to code accessible websites affects every aspect of the design process.

     

    For example, many developers do not know key concepts and terms that are essential to learning accessible wed design. Such concepts and terms are the Accessibility Tree, Semantics, and Focus. Each of these concepts builds off one another and it is from knowing them that a developer will be able to properly make accessible websites.

     

    Additionally, most web developers are not familiar with using a keyboard and a screen reader to navigate through a website. Therefore, these developers would not know how audio user interfaces work on their websites and how it differs from the visual user interfaces they regularly make. This makes sense because most web developers are designing their websites first for users who can interact with their site visually.

     

    Consequently, the first step towards being able to make accessible websites is to become knowledgeable about the subject. This is easier said than done which is why these concepts will be cover more extensively in other blogposts. However, once the learning curve is overcome, the most difficult barrier to making accessible websites will be overcome and a developer will more easily be able to make them.  

     

    Poor Structure

     

    The next pitfall that can trip up developers is poor document structure for webpages. The reason why is because if the Document Object Model (D.O.M) of an HTML webpage is not correctly organized it can complicate navigating through a site with a screen reader.

     

    A situation where this can happen is when elements in the D.O.M are not placed in the exact order as they appear visually because a CSS style has been applied to them. The example bellow illustrates code that would create a confusing experience for a user navigating a site by keyboard.

     

    
<!DOCTYPE html>
<html>
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device=width">
  <title>D.O.M Elements Not Ordered as They Appear Visually</title>
</head>
<body>
  <button>Add Food</button>
  <button style="float: right;">Buy Meal</button>
  <button>Add Condiments</button>
</body>
</html>

     

    The following before and after images shows how a browser would render the three buttons with and without the CSS Style.

     

     

    In order to create a logical checkout sequence, the developer applied a CSS style of “float: right;” so that the Buy Meal button is at the end. However, the button sequence of Add Food, Add Condiments, and Buy Meal is still not in that order for keyboard navigating users. This is because the code and the Accessibility Tree created from the code are not in the order of Add Food, Add Condiments, and Buy Meal.

     

    As a result, although the button sequence makes sense visually, the same cannot be said when moving between these buttons with assistive technology. For example, if a user is using the Tab keys to navigate from button to button, they would first land on the Add Food button, then jump to the Buy Meal button, and after that go to the Add Condiments button. Understandably, someone would be confused by this experience.

     

    A solution to fix the problem and keep the style would be to edit the code and move the Buy Meal button after the Add Condiments button. The following code shows the solution:

     

    
<!DOCTYPE html>
<html>
<head>
   <meta charset="utf-8">
   <meta name="viewport" content="width=device=width">
   <title>D.O.M Elements Not Ordered as They Appear Visually</title>
</head>
<body>
   <button>Add Food</button>
   <button>Add Condiments</button>
   <button style="float: right;">Buy Meal</button>
</body>
</html>

     

    Also, the developer could have in the first place rearranged the D.O.M order of the <button> elements like in the previous code example so that they can accomplish the same task without a CSS Style.

     

    Regardless of the approach, the new D.O.M structure is better because it is all round more cleaner and creates the same experience for all users. 

     

    HTML5 Semantic Elements

     

    Another pitfall is not using HTML5 semantic elements in situations where they can be used. One of the things that HTML5 added was new semantic elements. A semantic element clearly describes what the HTML element does both to the browser and the developer.

     

    Examples of semantic HTML elements are the <header>, <body>, <main>, <footer>, <form>, <nav> and <button> elements. Also, examples of non-semantic HTML elements are the <div> and <span> elements which do not give any information about what is inside of it.

     

    The reason why more semantic elements were added to HTML5 was because web developers would usually use non-semantic elements with semantic ID attributes to create an their own versions of the elements since they did not exist. For instance, a website would have the following HTML code of <div id="nav">, <div class="header">  and <div id="footer">.

     

    However, for the purposes of accessible web development, the new HTML5 semantic elements also adds functionality behind the scenes for assistive technologies. As a result, it is better to utilize these built-in assistive features in HTML5 because it will save a lot of work in making an accessible website.

     

    A situation where a developer should use a semantic element over a non-semantic is when a button is being made. A button is a unique HTML element that it is used to initiate other interactions on a webpage. However, some web developers prefer not to use the <button> element and instead like making a custom button with a <div> element.

     

    The problem with making a custom button is that it does not receive autofocus for keyboard users by browsers. Although a developer could make a custom button auto-focusable with the *tabindex* attribute, there are more functionalities that would need to be added as well in order to make a fully accessible element. Therefore, the amount of effort put into making a custom button truly accessible will bring diminishing returns. This is because the same objective can more easily be achieved with using the original <botton> element and styling it with CSS to make the button look like a <div> element.

     

    The point is that a developer should try to take advantage of semantic elements whenever they can since it will make accessibility easier.   

     

    Text Alternatives

     

    In addition, web developers sometimes do not add text alternatives to images and know how to properly use them. The practice of adding text alternatives is one of the simplest ways to make a webpage accessible.

     

    A text alternative is descriptive text that is added to an image that conveys contextual information about the image. This allows images to become accessible to screen reader users because the alternative text is read to them. Text alternatives are added to images by using the alt attribute on an <img> element. When writing alternative text for images, it is best to try to accurately describe what the image is with the alt attribute.

     

    The code below shows an alt attribute that has been added to the corresponding image.

     

    
<img src="Images/Christmas-Tree.jpg" alt="A lit Christmas Tree at night">

     

     

    In this case, the alt attribute of “a lit Christmas Tree at night” for the image will be read out when encountered by a screen read.

     

    If an alt attribute is not assigned to an <img> element, a screen reader will read the whole image file name. This can be useful at times if the file name describes the image. However, usually file names for images are random computer generated characters that do not give information about the content of an image. Therefore, it is a good practice to always add an alt attribute to an <img> element because it helps disabled users and is simple to apply.  

     

    Also, if an image is just being used for a decoration on a webpage and holds no significance, then an empty alt attribute should be applied on the image. The reason why is because we want the screen reader to not go to its default setting when seeing an <img> element without an alt attribute and the file name. Instead an applied empty alt attribute will result in a screen reader saying “image” when it comes across the image. As a result, the alt attribute should be used whenever possible.

     

    Lastly, there are other text alternative attributes that help assistive technologies. These attributes are A.R.I.A (Accessible Rich Internet Applications) specs that are specifically designed to make an HTML element accessible friendly. Examples of them are the role , aria-label , and the aria-labelledby attributes. However, what these attributes do and what A.R.I.A does is complex and so they will be covered in another article.

     

    The main takeaway is that developer should add alternative text with their images. This is because adding alternative text with attributes is easy to do and it does not affect a developer’s visual design. Also, the alternative text goes a long way for disabled users. Finally, alternative text like the alt attribute can enhance the user experience for regular users too by it being displayed in the event an image does not load.       

     

    Checking Code Accessibility

     

     

    The last pitfall that web designer fall into is not checking if their code is accessible and not knowing how to check it. There is no better way to see if a website is accessible than navigating a website with assistive technology.

     

    However, many web developers are not familiar with assistive technologies, like screen readers, and so they do not check their sites with them. The solution then for web developers is to familiarize themselves with screen readers. This can be done by downloading a screen reader and by learning the keyboard commands for it.

     

    As soon as developers overcome the learning curve for screen readers, they will to find accessibility issues in their websites. Also, it is a great way to get a feel for how disabled users navigate a website and that experience is invaluable. The reason why is because many web developers approach designing a website from a visual viewpoint. So developers getting an idea how a site is used non-visually will help a them see how to create accessible websites.

     

    A developer checking a website with a screen reader is a guaranteed way to make sure a website is accessible for all users.

    The first step to do this is to learn how screen readers and how other assistive technologies work. The exact commands and ways to navigate a site with a screen reader require explanations and examples. Therefore, in a future article, screen reader commands and other tricks will be explored.
